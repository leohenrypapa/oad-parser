# Target Environment Validation Checklist

This checklist defines target-environment validation for the OAD parser Sprint 2 live parser foundation before customer operational handoff.

This checklist is documentation-only. It does not execute target validation, automate privileged deployment, create site-specific Filebeat/Elastic Agent configuration, change parser/runtime behavior, implement one-hour 6100 PPS mode, or add radar semantics.

## Scope and assumptions

Target environment assumptions:

- Oracle Linux Server 9.6 is the target operating system.
- Python 3.9.2 is the target Python runtime.
- Live capture and systemd operation require root runtime or an approved equivalent capability model.
- Customer handoff uses the customer runtime/operator handoff pack generated by `scripts/make_customer_pack.sh`.
- The customer pack must pass `scripts/validate_customer_pack.py` before target handoff.
- Filebeat/Elastic Agent 8.17.3 is the expected SIEM handoff assumption.
- Final Filebeat/Elastic Agent version and site configuration must be confirmed by the SIEM owner.
- Pass/fail live validation applies only to connected ECG interfaces.
- `eno1` through `eno5` are documented interface examples, not a requirement that every interface exists or carries ECG traffic.

Out of scope for this checklist:

- Do not execute target validation from this repo.
- Do not automate privileged target deployment.
- Do not create site-specific Filebeat/Elastic Agent configuration.
- Do not require all `eno1` through `eno5` interfaces to pass if they are not connected ECG interfaces.
- Do not commit target logs or runtime outputs.
- Do not commit real PCAPs, raw operational payloads, secrets, local reports, or runtime outputs.

## Evidence handling

Recommended local evidence staging paths on the operator workstation:

- `reports/target/target-environment-validation.md`
- `reports/target/target-environment-validation.json`
- `reports/target/systemd-validation.md`
- `reports/target/interface-validation.md`
- `reports/target/storage-validation.md`
- `reports/siem/handoff-validation.md`

These evidence paths are examples for sanitized local evidence. Generated evidence is not committed by default.

The following evidence must not be committed:

- target logs with site-sensitive details
- real PCAPs
- raw operational payloads
- `/nsm/ecg` runtime outputs
- hostnames
- IP addresses
- SIEM endpoints
- tokens
- certificates
- private keys
- index names
- customer-specific interface mappings
- systemd journal exports unless sanitized and approved

If evidence must be shared, sanitize it first and confirm it is approved for the intended audience.

## Phase 1 - Target host prerequisite check

Owner role: target operator or production maintainer.

Commands to run on target:

    cat /etc/os-release
    python3.9 --version
    id
    sudo -n true || echo "sudo-interactive-or-unavailable"
    uname -a

Pass criteria:

- Target host is Oracle Linux Server 9.6 or approved equivalent for the release event.
- Python reports Python 3.9.2.
- Operator has approved root or sudo access for live capture/systemd validation.
- Evidence is recorded without committing hostnames, IP addresses, usernames, or other site-sensitive values.

Fail criteria:

- Target OS is not confirmed.
- Python 3.9.2 is unavailable.
- Root runtime or approved capability model is unavailable for live capture/systemd validation.

## Phase 2 - Customer pack install and copy check

Owner role: release engineer and target operator.

Commands to run before target copy:

    bash scripts/make_customer_pack.sh /tmp/oad-parser-customer-runtime.tar.gz
    .venv/bin/python scripts/validate_customer_pack.py --pack /tmp/oad-parser-customer-runtime.tar.gz --output-json /tmp/oad-customer-pack-validation.json
    grep -n '"status": "passed"' /tmp/oad-customer-pack-validation.json

Target copy checklist:

- Copy only the approved customer runtime/operator handoff pack.
- Do not copy internal engineering source packs unless specifically approved.
- Confirm the customer pack includes `CUSTOMER-PACK-MANIFEST.json`.
- Confirm target extraction location is approved by the customer/operator.

Pass criteria:

- Customer pack validates with `"status": "passed"`.
- Pack is copied using an approved transfer method.
- No internal/dev-only source-pack artifacts are used as the default customer runtime package.

Fail criteria:

- Customer pack validation fails.
- Customer pack is missing `CUSTOMER-PACK-MANIFEST.json`.
- Internal engineering source pack is substituted without explicit approval.

## Phase 3 - Target configuration check

Owner role: target operator.

Configuration path:

    /etc/oad-parser/ecg_conf.ini

Example source file:

    config/ecg_conf.example.ini

Checklist:

- Create `/etc/oad-parser`.
- Install or adapt `config/ecg_conf.example.ini` as `/etc/oad-parser/ecg_conf.ini`.
- Confirm selected interface is a connected ECG interface.
- Confirm output paths point under `/nsm/ecg`.
- Confirm rotation, pruning, high-water, and critical threshold settings are site-appropriate.
- Do not commit the target config if it contains site-sensitive values.

Suggested commands:

    sudo install -d -m 0755 /etc/oad-parser
    sudo install -m 0644 config/ecg_conf.example.ini /etc/oad-parser/ecg_conf.ini
    sudo sed -n '1,200p' /etc/oad-parser/ecg_conf.ini

Pass criteria:

- `/etc/oad-parser/ecg_conf.ini` exists.
- Config points to the intended connected ECG interface.
- Config points to `/nsm/ecg` output paths.
- Site-sensitive values are not copied back into the repo.

Fail criteria:

- Config path is missing.
- Config references the wrong interface.
- Config references unapproved output paths.

## Phase 4 - `/nsm/ecg` directory, ownership, and permissions

Owner role: target operator.

Required output directory:

    /nsm/ecg

Expected parser-owned outputs:

    /nsm/ecg/ecg-current.json
    /nsm/ecg/ecg-audit.jsonl
    /nsm/ecg/ecg-status.json

Suggested commands:

    sudo install -d -m 0755 /nsm/ecg
    sudo ls -ld /nsm /nsm/ecg
    sudo test -d /nsm/ecg

Pass criteria:

- `/nsm/ecg` exists.
- Ownership and permissions allow the root-running parser/systemd service to write outputs.
- Permissions are no broader than required for local operator and SIEM collection needs.

Fail criteria:

- `/nsm/ecg` is missing.
- Parser cannot write to `/nsm/ecg`.
- Permissions expose runtime outputs unnecessarily.

## Phase 5 - Interface discovery and connected ECG interface selection

Owner role: target operator and mission/site network owner.

Documented interface examples:

    eno1
    eno2
    eno3
    eno4
    eno5

Suggested commands:

    ip link show eno1 || true
    ip link show eno2 || true
    ip link show eno3 || true
    ip link show eno4 || true
    ip link show eno5 || true

Selection rules:

- Identify which of `eno1` through `eno5` are present.
- Identify which present interfaces are connected ECG interfaces.
- Run live validation only against connected ECG interfaces.
- Do not treat non-existent or non-ECG interfaces as failed parser validation.
- Do not commit customer-specific interface mappings.

Pass criteria:

- At least one connected ECG interface is identified for live validation, when live validation is in scope.
- Pass/fail live validation applies only to connected ECG interfaces.
- Non-connected interfaces are marked not applicable.

Fail criteria:

- No connected ECG interface is identified when live validation is required.
- Validation is attempted against a known non-ECG interface and treated as production evidence.

## Phase 6 - `oad_parser live` smoke test

Owner role: target operator.

Run a bounded live smoke on the connected ECG interface only.

Example for `eno1`:

    sudo python3.9 -m oad_parser live --config /etc/oad-parser/ecg_conf.ini --interface eno1 --max-frames 1

Equivalent examples for other connected interfaces:

    sudo python3.9 -m oad_parser live --config /etc/oad-parser/ecg_conf.ini --interface eno2 --max-frames 1
    sudo python3.9 -m oad_parser live --config /etc/oad-parser/ecg_conf.ini --interface eno3 --max-frames 1
    sudo python3.9 -m oad_parser live --config /etc/oad-parser/ecg_conf.ini --interface eno4 --max-frames 1
    sudo python3.9 -m oad_parser live --config /etc/oad-parser/ecg_conf.ini --interface eno5 --max-frames 1

Pass criteria:

- Command starts without Python/runtime import failure.
- Command can access the selected connected ECG interface.
- Bounded run exits cleanly when `--max-frames` is satisfied or records an expected no-frame condition if no traffic is present.
- No real payload data is copied into the repo.

Fail criteria:

- Python import/runtime failure.
- Permission failure on raw socket access.
- Wrong interface selected.
- Output path write failure.

## Phase 7 - Systemd template validation

Owner role: target operator.

Template source:

    deploy/systemd/ecg-parser@.service

Install location:

    /etc/systemd/system/ecg-parser@.service

Install and reload commands:

    sudo install -m 0644 deploy/systemd/ecg-parser@.service /etc/systemd/system/ecg-parser@.service
    sudo systemctl daemon-reload
    systemctl cat ecg-parser@.service

Instance examples:

    ecg-parser@eno1.service
    ecg-parser@eno2.service
    ecg-parser@eno3.service
    ecg-parser@eno4.service
    ecg-parser@eno5.service

Start/status/stop examples for `eno1`:

    sudo systemctl start ecg-parser@eno1.service
    sudo systemctl status ecg-parser@eno1.service --no-pager
    sudo systemctl stop ecg-parser@eno1.service

Equivalent instance names for other connected ECG interfaces:

    sudo systemctl start ecg-parser@eno2.service
    sudo systemctl status ecg-parser@eno2.service --no-pager
    sudo systemctl stop ecg-parser@eno2.service

    sudo systemctl start ecg-parser@eno3.service
    sudo systemctl status ecg-parser@eno3.service --no-pager
    sudo systemctl stop ecg-parser@eno3.service

    sudo systemctl start ecg-parser@eno4.service
    sudo systemctl status ecg-parser@eno4.service --no-pager
    sudo systemctl stop ecg-parser@eno4.service

    sudo systemctl start ecg-parser@eno5.service
    sudo systemctl status ecg-parser@eno5.service --no-pager
    sudo systemctl stop ecg-parser@eno5.service

Pass criteria:

- `systemctl daemon-reload` completes.
- Template is visible through `systemctl cat`.
- Instance starts on a connected ECG interface.
- Instance status is active or exits in an expected bounded/smoke condition.
- Instance can be stopped cleanly.

Fail criteria:

- Template install fails.
- `systemctl daemon-reload` fails.
- Service cannot access Python 3.9.2, config, interface, or `/nsm/ecg`.
- Service loops rapidly or fails restart-limit behavior.

## Phase 8 - Output file checks

Owner role: target operator.

Expected files:

    /nsm/ecg/ecg-current.json
    /nsm/ecg/ecg-audit.jsonl
    /nsm/ecg/ecg-status.json

Suggested commands:

    sudo ls -l /nsm/ecg
    sudo test -f /nsm/ecg/ecg-current.json || true
    sudo test -f /nsm/ecg/ecg-audit.jsonl || true
    sudo test -f /nsm/ecg/ecg-status.json || true
    sudo tail -n 5 /nsm/ecg/ecg-current.json || true
    sudo tail -n 5 /nsm/ecg/ecg-audit.jsonl || true
    sudo cat /nsm/ecg/ecg-status.json || true

Behavior expectations:

- `/nsm/ecg/ecg-current.json` contains JSON Lines records despite the `.json` suffix.
- `/nsm/ecg/ecg-audit.jsonl` contains JSON Lines audit records.
- `/nsm/ecg/ecg-status.json` is local operator status unless later central collection is approved.
- File absence may be acceptable when no frames have been processed, but the reason must be documented.

Pass criteria:

- Output behavior matches the live parser run conditions.
- JSON Lines output is parseable line by line.
- Status snapshot is local operator evidence.
- Runtime outputs are not committed.

Fail criteria:

- Parser emits invalid JSON Lines for records it writes.
- Audit output is missing when audit events are expected.
- Status output is invalid or inconsistent with service state.
- Runtime outputs are copied into the repo.

## Phase 9 - Storage high-water and critical threshold validation

Owner role: target operator and production maintainer.

Use controlled/safe validation only. Do not fill production disks as a test.

Validation approach:

- Review config thresholds in `/etc/oad-parser/ecg_conf.ini`.
- Use a test filesystem, temporary mount, or controlled lab host if threshold behavior must be exercised.
- Confirm high-water behavior blocks or prunes only as designed.
- Confirm active-file protection is preserved.
- Confirm critical threshold behavior stops or fails safely as documented.

Pass criteria:

- High-water behavior is verified in a controlled/safe environment.
- Critical threshold behavior is verified without endangering operational storage.
- Evidence documents method and limitations without site-sensitive values.

Fail criteria:

- Validation risks production storage exhaustion.
- Active file is deleted unexpectedly.
- Parser continues writing beyond critical threshold.

## Phase 10 - Filebeat/Elastic Agent handoff validation

Owner role: SIEM owner and target operator.

Expected assumption:

- Filebeat/Elastic Agent 8.17.3 is the expected handoff assumption.
- Final Filebeat/Elastic Agent version and site configuration must be confirmed by the SIEM owner.

Boundary:

- Parser owns `/nsm/ecg` outputs.
- SIEM owner owns endpoint, index, certificate, token, pipeline, enrollment, and deployment configuration.
- Parser repo does not commit site-specific SIEM configuration.

Parser-owned files for SIEM consideration:

    /nsm/ecg/ecg-current.json
    /nsm/ecg/ecg-audit.jsonl

Local-only unless later central collection is approved:

    /nsm/ecg/ecg-status.json

Pass criteria:

- SIEM owner confirms Filebeat/Elastic Agent version and site configuration.
- SIEM owner confirms the final collection path and parser-owned file boundary.
- No endpoint, index name, certificate, token, key, hostname, or IP address is committed.
- SIEM evidence is sanitized before sharing.

Fail criteria:

- SIEM handoff assumes unconfirmed endpoint/index/cert/token values.
- Site-specific Filebeat/Elastic Agent configuration is committed.
- `/nsm/ecg/ecg-status.json` is centrally collected without approval.

## Final target validation summary

A target validation summary should record:

- Target validation date.
- Oracle Linux Server 9.6 confirmation.
- Python 3.9.2 confirmation.
- Customer pack validation status.
- Config path validation status.
- `/nsm/ecg` path and permission validation status.
- Connected ECG interface used for validation.
- `oad_parser live --max-frames` smoke result.
- Systemd instance validation result.
- Output file validation result.
- Storage behavior validation result or reason deferred.
- Filebeat/Elastic Agent 8.17.3 assumption and SIEM owner confirmation status.
- Known limitations.
- Sanitization review outcome.

This summary must not include hostnames, IP addresses, SIEM endpoints, tokens, certificates, private keys, index names, target logs with site-sensitive details, customer-specific interface mappings, real PCAPs, raw operational payloads, or `/nsm/ecg` runtime outputs unless sanitized and approved.
